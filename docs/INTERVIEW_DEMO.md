# 三分鐘面試展示腳本

## 0:00–0:30 問題與目標

「這個專案原本是 MVC 裝備管理系統。我把它延伸成 ARAM Mayhem AI 推薦服務，解決玩家面對英雄、四個海克斯階段與裝備搭配時，很難快速判斷最佳方向的問題。」

開啟首頁，指出這不是單純的聊天介面，而是包含英雄、海克斯、裝備與玩家回饋的完整系統。

## 0:30–1:10 推薦流程

1. 進入 AI 推薦。
2. 用滑鼠選擇英雄。
3. 選擇一個後期階段，展示先前已選海克斯與當輪三選一。
4. 加入簡短對局狀況並產生推薦。

說明：「後續階段會參考先前選擇，不會把每一輪視為獨立問題；裝備由系統推薦，玩家不需要先知道答案。」

## 1:10–1:45 RAG 與可解釋資料

切到英雄與海克斯知識庫，展示：

- 英雄定位、核心裝備、情境裝備與人工打法筆記。
- 海克斯的稀有度、套裝系列與戰鬥屬性。
- 玩家收藏與採納結果如何成為下一次相似情境的檢索參考。

說明：「玩家回饋不會訓練第三方模型權重，而是以可控的資料形式加入 RAG 上下文。」

## 1:45–2:20 工程與資料庫

展示 README 架構圖並說明：

- Controller 依賴 Repository 與 Service，不直接寫 SQL。
- SQL Server 資料已遷移到 Supabase PostgreSQL。
- 12 張資料表啟用 RLS，密鑰只由伺服器端環境變數讀取。
- Docker 與 Render 提供一致的部署環境。

## 2:20–2:50 品質與資安

開啟 GitHub Actions 或 README 品質區塊：

- xUnit 測試核心規則與 MVC 安全慣例。
- 30 組離線 AI 評測案例防止禁用裝備、簡體或英文名稱混入推薦底線。
- CI 執行 secret scan、套件弱點、Supabase contract、Release build 與測試。
- `/healthz`、`/readyz`、local smoke 與 cloud smoke 用於部署驗證。

## 2:50–3:00 收尾

「這個專案讓我實際處理了 AI 格式不穩定、資料庫遷移、權限與密鑰管理、部署以及可重複驗證。我下一步希望把這些經驗帶進多人協作的正式開發流程。」

## 展示前檢查

- 線上網站與 `/readyz` 可正常開啟。
- 使用測試帳號，不展示管理員密碼或環境變數。
- 預先準備一組推薦輸入，避免現場等待免費模型。
- 若 OpenRouter 暫時無法使用，改展示已收藏案例並說明快取與降級策略。
- 不開啟 Supabase、Render 或使用者機密頁面。
