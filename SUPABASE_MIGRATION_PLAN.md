# Supabase PostgreSQL 遷移計畫

這份計畫的目標是把目前本機 SQL Server 專案，安全地搬到 Supabase PostgreSQL，
同時保留可回退路徑。

## 現況結論

目前專案大量使用 `Microsoft.Data.SqlClient`、`SqlConnection`、`SqlCommand` 和 T-SQL。
因此不能只把 `DefaultConnection` 換成 Supabase PostgreSQL 連線字串。

需要先把資料存取層從 SQL Server 專用語法拆出來，再逐步改成 PostgreSQL 可用語法。

## 建議分階段

### 第 1 階段：部署骨架

已建立：

- `Dockerfile`
- `.dockerignore`
- `render.yaml`
- `DEPLOYMENT_SECURITY.md`
- `Database:Provider` 設定
- 安全的連線字串摘要 log，不再用 SQL Server parser 解析所有資料庫字串

這一階段可以先讓網站用 Docker 打包，並在正式部署平台上用環境變數管理 key。

### 第 2 階段：資料存取層抽離

把 controller 裡直接開 `SqlConnection` 的程式搬到 service/repository。

優先順序：

1. Account / User：登入、註冊、個人資料。
2. AI Recommendation：推薦、收藏、使用者採納回饋。
3. LolAramGuides / LolAramAugments：英雄與海克斯知識庫。
4. Equipment / Calculator：裝備、套裝、公式紀錄。

### 第 3 階段：PostgreSQL 語法轉換

常見轉換：

| SQL Server | PostgreSQL |
| --- | --- |
| `IDENTITY(1,1)` | `GENERATED BY DEFAULT AS IDENTITY` |
| `NVARCHAR(MAX)` | `TEXT` |
| `BIT` | `BOOLEAN` |
| `GETUTCDATE()` | `NOW() AT TIME ZONE 'UTC'` |
| `TOP (@count)` | `LIMIT @count` |
| `[ColumnName]` | `"ColumnName"` or snake_case |
| `MERGE` | `INSERT ... ON CONFLICT ... DO UPDATE` |
| `IF COL_LENGTH(...) IS NULL` | `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` |

### 第 4 階段：Supabase 建表與 RLS

正式資料表至少需要：

- users
- user_activity_logs
- ai_recommendation_cache
- ai_recommendation_favorites
- equipments
- equipment_loadouts
- lol_aram_guides
- lol_aram_augments
- lol_aram_items
- lol_aram_augment_series
- lol_aram_synergy_rules

RLS 原則：

- 使用者只能讀寫自己的收藏、紀錄、裝備組合。
- 管理員才可寫入英雄、海克斯、裝備知識庫。
- `service_role` key 只放後端，用於管理任務，不給前端。

### 第 5 階段：資料搬遷與驗證

1. 從 SQL Server 匯出乾淨資料。
2. 匯入 Supabase staging project。
3. 對比筆數與關鍵欄位。
4. 用 Docker 啟動網站連 staging Supabase。
5. 驗證登入、AI 推薦、收藏、英雄、海克斯、裝備、公式。
6. 再切 production Supabase。

## 回退策略

- 每次遷移前保留 SQL Server 備份。
- Supabase 先用 staging project，不直接動 production。
- `Database:Provider` 預設保持 `SqlServer`，直到 PostgreSQL repository 完成。
- Docker 部署先驗證網站可啟動，再接正式資料庫。

## 下一個建議實作

下一步應從 `AccountController` 開始，把登入/註冊改成 repository。
這是正式部署最重要的一塊，因為登入會牽涉密碼雜湊、使用者權限和 RLS。
